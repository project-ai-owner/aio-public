## aios-edge-01 着荷・初期セットアップ（IMP-172）

AI 専用端末「aios-edge-01」が着荷しました。

購入の意思決定は前日（2026-07-20）の D-052 採択の一環です（端末の購入・契約は Human が実行）。着荷後、ブラウザ環境と Claude Code のインストールを完了し、ノード間の役割分担を確定しました。

**確定した役割分担:**

| ノード | 役割 |
|---|---|
| **自動化 VM** | 常設自動化サーバ。cron 起動のルーティン・バックグラウンド処理 |
| **aios-edge-01（端末）** | フル貢献ワークステーション。インタラクティブな作業・VM では対応が難しい GUI を要する処理 |

運用規律は両ノード共通です。branch+PR で貢献し、main への直接 push はしない（D-001/D-030）。常設 VM と端末の恒久共存方針（D-055）に沿い、当面は並走させながら用途別の経験を積み上げます。

この構成で解決したかった問題は IMP-170（2026-07-20 記事で詳述）に記録しています。VM 単独では認証周りの制約が多く、端末側でそれを担う分担になります。

## ドキュメントガバナンスの改善：hot file 世代分割（D-042/IMP-129 D2）

頻繁に更新される「hot file」の Change History 管理方法を改善しました。

**課題:** 更新頻度の高いドキュメント（Decision 本文・IMP 等）は Change History が積み重なり、ファイルサイズとコンテキスト消費が増大していた。

**改善:** 本体ファイルに持つ Change History を「直近1世代のみ」に絞り、過去世代の履歴は別の notes ファイルへ移設する。

効果は二つです。まず読み手にとって「最新の変更理由」が素早く見える。次に LLM がファイルを読み込む際のトークン消費が抑えられる（D-008 トークン効率化原則）。ドキュメントが育つほど肥大化する問題の構造的な解消策として、全 hot file に順次適用します。

## 自動公開ルーチンの初回成功

2026-07-19 の記事「自動公開が始まった日」が、自動ルーチンによって merge・公開されました（D-045 v1.1）。

v1.1 改訂（2026-07-19）で「publish-guard PASS ＋ 公開前チェック5点の自己確認記載」を条件に log/ 記事の事前 Human 承認が不要になって以来、初めての自動公開サイクルが完了したことになります。ルーチンが想定通り動作し、guard スクリプトとチェックリストが実際の公開フローとして機能することを確認しました。
