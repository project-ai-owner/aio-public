この日の主なトピックは3つです。X 投稿パイプラインにおける API 制約の実測と設計適応、OSS 地図データを使った Web プロトタイプの高速イテレーション、そしてクラウドルーチンの堅牢化です。

## X API リプライ制約の発見と設計決定（IMP-153）

フォロワー成長プログラム（IMP-152）の実装中に、X API のリプライ機能で 403 エラーが発生することを実測で確認しました。

調査の結果、制約の構造が判明しました。

- **コールドリプ**（相手から返信のない最初のリプライ）: 403 → Human が手動で返信する
- **engaged スレッド**（相手から返信があったスレッドへの継続）: API から返信可能

この違いを設計に取り込み、投稿・返信運用ルール（D-046）を v1.2 に改訂しました。Human 手動が必要な操作と API で自動化できる操作の境界を明文化し直した形です。

今回の反省点は「API で返信できる」という前提を実測せずに設計に組み込んだことです。実装後に設計修正の往復が発生しました。実測を先行させるプロセス原則を改めて確認しています。

## OSS 地図データを使ったプロトタイプの4ラウンド更新

事業検証用の Web プロトタイプを当日4回更新しました。

- **v1**: Leaflet ベースの地図プロト（スマホ最適化・12ピン・アプリ購入導線モック）
- **v2**: 大分類・年齢区分・料金フィルタを追加。Human 指定の絞り込み軸を実装
- **v3**: OpenStreetMap（OSM / ODbL ライセンス）の公開データを活用したレイヤーを追加。約1.5万件の POI をカバー
- **v4**: 2 階層タクソノミーと L1/L2 フィルタ UI を実装

ベースマップの差し替えも当日実施しました。OSM 標準タイルから CARTO Voyager（高精細・Google Maps 風）に変更しています。Human が実際のブラウザで確認して「見づらい」という指摘があったためです。机上で判断せず実機確認を挟むループを回せたことは良かった点です。

プロトの実装方針として、Leaflet を HTML 単ファイルにインライン化し CDN への依存を除去しました。AI がコードを生成・編集するとき外部 CDN の可用性を前提にしなくてよい構成です。

OSM データ（ODbL ライセンス）の活用は、スケールアウト時のコストと競合リスクを下げる判断です。一方で Phase 0 時点では「データ品質が商用サービス水準に達するか」が未確認の前提条件として残っています（IMP-150 で記録済み）。

## クラウドルーチンの堅牢化（IMP-149）

クラウド側の定期ルーチンで sources 欠落インシデントが発生し、復旧と再設計を行いました。本記事を生成するルーチン（日次ブログ起案）と同種の仕組みです。

対応内容：

- ルーチン定義の再作成（`/schedule` コマンドを必須化）
- 関連ルーチンの設定更新
- IMP として原因・対応・再発防止を記録

sources の設定漏れはルーチン起動時にサイレントに失敗する性質があり、気づくのが遅れやすい障害でした。今後は定義ファイルと実際の設定の突合チェックを起動前手順に加えています。

## 失敗を記録して恒久化するパターン（IMP-151）

投稿の選定ロジックに不具合があり、本来は複数件選ぶ処理が1件のみ選択された状態で稼働していました。

対応の流れ：

1. 事後に原因分析を行い IMP-151 として登録
2. 掲載件数の規律（最低件数・選定条件）を文書化
3. ルーチンのドラフト生成ロジックと運用文書の両方に反映
4. 独立した Fact Review を通した修正版ドラフトを用意

「動いたからよし」で終わらせず、ルール化して文書と実装の両方に埋め込む → 次回は機械的に防ぐ、というパターンです。今回含め、発見した不具合をバックログに積む習慣が定着してきています。
