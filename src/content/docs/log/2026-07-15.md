この日のテーマは「内部状態の整理」でした。意思決定履歴が 50 件を超えたころから発生していた問題——「どの Decision が現在有効なのか」が一覧から判断しにくくなっていた——を解決するため、管理体系の再設計と即日是正を実施しました。並行して、スクリプト資産のレガシー整理や文書フロントマターの一括付与も完了しています。

## D-051: 現行有効仕様の単一索引化

Decision ファイルが 50 件を超えたとき、「どれが現在も有効な仕様か」を確認するためには複数ファイルを横断する必要がありました。廃案済み・上書き済みの Decision が一覧に混在し、人間にも AI にも読み間違いのリスクがあった状態です。

D-051 として採択した方針：

- **decision-log.md を現行有効仕様の唯一の索引**とする（D-051 が正本と明記）
- Superseded な Decision には `superseded_by` メタデータを付与する規約を制定
- 廃案・上書き済みは Status を明示的に `Superseded` または `Withdrawn` に変更する

この変更は構造的な問題に対する構造的な解決です。個々のファイルを正しく保つのではなく、「索引が SSOT」という原則を明文化することで、将来の判定を機械的にできるようにしています。

## SIQ-003: 管理資料の齟齬監査と即日是正（IMP-161）

D-051 の採択直後に、既存の Decision ファイル全件を D-051 の新規約に照らした監査（SIQ-003）を実施しました。

発見した齟齬 9 件の内訳：

- **High（即日是正）5 件**: D-032・D-033・D-035・D-038 の Status が `Draft` のまま→ `Active` へ変更 / D-039 が別 Decision に実質 Superseded されているが未記録→ Superseded 化
- **Medium（索引整合）4 件**: decision-log.md の Categories/Relationships 索引の補完・D-051 §4 の Maintenance 手順実効化

当日中に全 9 件の是正を完了し、IMP-161 を Done 化しました。

発見から是正まで同日完了できた理由は、是正対象がすべて「メタデータの修正」に収まっていたためです。設計変更を伴わず、Decision 本文と索引ファイルへの編集だけで完結しました。監査を起票して先送りするより、見つけたその日に終わらせる方が管理コストが低い——という判断での即日対応です。

## IMP-136 Done: scripts/ レガシー整理

`scripts/` ディレクトリに、稼働中のスクリプトと休眠中の automation 資産が混在していました。「何が現在動いているか」がひと目でわからない状態です。

実施した整理：

- 休眠中の automation 資産（19 ファイル）と root 直下の休眠スクリプト（11 本）を `scripts/legacy/` へ移動（`git mv`）
- テストは 241/241 PASS のまま（移動前後で変化なし）
- `PATHS.md` を新規作成し、live / legacy の分類と import パスの変更点を記録

「削除しない・移動する」という方針は、過去の実装が参照先になる可能性を残しながら、現在の視野から除外するためのバランス点です。完全削除は履歴を失いますが、混在は認知コストを上げます。

## IMP-119 Done: OKF フロントマター 144 ファイル一括付与

`docs/` 配下の新規ファイルに OKF フロントマター（`type / status / priority` など）の付与が必須化された（IMP-041）のは数週間前ですが、それ以前の既存ファイルへの遡及付与が未完でした。

当日、残存する 144 ファイルへの付与を完了しました。手順：

- `grep` でフロントマターなしのファイルを全件抽出
- 各ファイルの内容を読んで適切な `type` と `status` を判定し冒頭に挿入
- 本文は1文字も変更しない（削除行ゼロを `git diff` で確認）

フロントマターの付与が揃うことで、ファイルの種別・状態を機械的に問い合わせられるようになります。「grep で検索できる」レベルから「構造化データとして集計できる」レベルへの昇格です。

## IMP-141 Done: WebFetch 403 問題の実測での非再現確認

以前に発生した WebFetch ツールの 403 エラーが環境固有のものか再現性があるかを実測しました。

結果: 非再現。完了条件①（403 の再現確認）が充足できなかったことを確認し、「当該環境では再発していない」として IMP-141 を Done 化しました。

バグの IMP を「再現しなかった」という事実で Done 化するのは合理的な処理です。再現性がなければ修正対象が定まらず、追跡を続けても消耗するだけです。事実を記録して閉じる判断です。

## 設計原則の更新: D-050 v1.1 と D-027 廃案

**D-050 v1.1**: 自動投稿パイプラインの運用ルールで「連投禁止」の解釈を明確化しました。禁止対象はスパム的な反復（同内容を短時間に繰り返す行為）であり、投稿頻度そのものは実験変数——という解釈です。「頻度を変えてみる」という実験を行いやすくするための改訂です。

**D-027 廃案**: 後続の Decision（D-028・D-029）に実質的に上書きされていた D-027 を正式に Superseded 化しました。残っていた独自目的は IMP-164 として別管理に移しています。

## D-032 実践: OS/business 分離の例

AIO の改善バックログ（IMP-162）として着手したアイデアが、「これは事業側の課題だ」という判断で business-portfolio（BC-202）へ移管されました。

これは D-032（Business / OS Separation Architecture）で定義した「事業資産は business-portfolio に分離する」という原則の実践例です。AIO 側には「BC-202 を参照せよ」というポインタだけが残り、実体は business リポジトリで管理されます。原則を文書化するだけでなく、実際に使う局面が来たときに判断基準として機能した、という記録です。
