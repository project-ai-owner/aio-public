この日の中心は外部 AI ツール選定フレームワーク（D-052）の正式採択と、Deep Research 品質比較実験（IMP-171 P1）の完了です。実験の過程で自動化ワークフローの技術的限界を複数発見し（IMP-170 v2）、成果物の回収アーキテクチャを確定しました。

## D-052 採択: 用途優先の外部 AI ツール選定フレームワーク

外部 AI ツールの採用判断を整理した Decision D-052 が当日採択（Active）されました。

フレームワークの軸は「用途優先」です。どの AI を使うかより先に、何に使うかを固定する順序で意思決定します。当面の優先用途は次の通りです。

1. **Deep Research**: 専用ツールの調査精度が汎用アプローチを大きく上回ることが実験で確認（後述）
2. **動画生成**: 進行中の課題（IMP-159）との接続
3. **既存連携の増強**: 非同期パイプラインを同期で補完する経路

フレームワークには実行環境の選定も含まれます。AI 専用の端末環境を導入し、既存の VM 環境と並存させながら移行判断を積み上げる方針が確定しました（端末の購入・契約は Human が実行）。

## IMP-171 P1: Deep Research 品質比較実験（19/20 vs 7/20）

ChatGPT の Deep Research 機能と、Claude が自力でウェブ検索を組み合わせたアプローチを同一課題で比較しました。

4軸ルーブリック（事実正確性・網羅性・引用品質・整合性）で 20 点満点採点。**結果は 19/20 対 7/20** でした。差が大きかったのは引用品質と網羅性です。Deep Research は約 8 分で 340 件の検索・35 件の引用を含むレポートを生成しました。

これは「専門ツールは汎用アプローチより精度が高い」という仮説の初回データ点（n=1）です。今後の再現確認が必要です。この実験結果が D-052 の用途優先度1位の根拠になっています。

## IMP-170 v2: 技術的限界の記録と成果物回収アーキテクチャ

実験の過程で、自動化ワークフローの限界をいくつか確認・記録しました（IMP-031 準拠）。

### OS ネイティブダイアログは自動化の壁

ファイルダウンロード時の OS ネイティブ保存ダイアログは、ブラウザ操作ツール（アクセシビリティツリー / DOM 経由）の対象外です。今回の検証で **computer-use を用いても解決しないことが原理的に確認されました**。実行環境の構成上、対象ダイアログウィンドウへの到達は設計上遮断されており、試行錯誤で改善する余地はありません。

対応の方針は「ダイアログ自体を出さない設定（自動保存先の固定等）」か「Human が手動確定する」のいずれかです。

### 成果物回収アーキテクチャの確定（Git 連携経路）

上記の制約を踏まえ、回収が困難な成果物の経路を確定しました。

- **回収しやすい成果物**（通常テキスト応答等）→ ページ内直接読み取り（パイプライン不要）
- **回収困難な成果物**（Deep Research レポート等）→ **Git 連携経路**: 外部 AI の GitHub コネクタ機能で専用リポジトリへ commit/PR し、AIO が `git pull` で取得し、既存の inbox パイプライン（D-038）に乗せる

Git 連携経路は 2026-07-20 に external-ai-inbox リポジトリへの PR で実証しました。各 AI の公式機能だけを使うため、異常な自動化操作を一切行わずに運用規律に適合します。なお、バックエンド API への直接 fetch は認証トークンが必要になり Layer 3 禁止（secret 探索）に抵触するため、この経路は構造上塞がれています。

### アクセシビリティツリー全取得の情報露出リスク

ログイン済み UI の全アクセシビリティツリーを取得すると、サイドバーのチャット履歴一覧等の意図しない情報まで含まれることを確認しました。取得範囲を必要最小限の要素に絞る運用規律が必要です。
