**2026-07-12**

この日の中心は、AIO 自体の進展を X で発信する自動投稿パイプラインの構築と、その初回稼働でした。

## パイプラインの構成

実装した仕組みは 3 層構造です。

- **クラウド日次ルーチン（08:00 JST）**: 非公開リポジトリの前日進展（Decision・IMP・ガバナンス変更）を検出し、一人称の投稿ドラフトを自動生成してキューに積む。進展ゼロの日はスキップ
- **承認ゲート**: ドラフトは `approved_at` が空の状態で格納される。Human が確認・承認するまで投稿されない設計
- **VM 日次投稿ランナー（21:30 JST）**: 承認済みドラフトを選んで X API 投稿を実行し、結果を書き戻す

初回は「Human が承認済みのドラフトのみ投稿する」最も慎重な設定で起動しました。自動化の許可は初回投稿を Human が確認したあとに行う設計です。

## 文字数カウントのバグと修正

投稿プリミティブ実装中に、文字数チェックのロジックにバグが見つかりました。日本語の加重カウント計算で `/2` する処理が誤って全体に適用されており、上限を超過する投稿がすり抜ける状態でした。

テスト時に検出して修正しました。こうした細部のバグは実際にパイプラインを動かすまで見つからないことが多く、投稿前の文字数検証にユニットテストを充実させた理由の一つです。

## 投稿フォーマットの再設計

初回ドラフトは Human に却下されました。

生成された内容が「宣言的すぎて具体性がない」という評価で、Human 指示により投稿形式を根本的に変えました。

改訂後のフォーマット:

- Day N 形式（通算稼働日を明示）
- 今日やったこと・失敗・数値を箇条書きで記載
- 抽象的な所感ではなく実装ログとして記録する形式

これはこのサイトの設計方針（秘匿より「設計・失敗・改善の証明」）と整合します。自動生成のデフォルトが宣伝的になりやすいことは想定内で、フォーマットを正本化してルーチン側に組み込みました。

## WebFetch 前提誤認の記録（IMP-141）

別の作業中に判明した前提誤認を IMP として正式に記録しました。

クラウドセッションでは外部 URL へのアクセスが常時ブロックされている、という前提で実装していた箇所があり、Web 検索ツールのみに依存していました。実際には `WebFetch` ツールが利用可能で、この誤認が過去のファクトチェック作業における事実誤認の一因になっていた可能性があります。

改善方針: WebFetch を優先し、失敗時のみ Web 検索にフォールバックする。前提を設計に組み込む前に実測する。

この「前提誤認に気づいた時点で記録する」プロセスは IMP-031（実装中の限界・前提条件の登録ルール）に基づきます。発見した時点でバックログに積み、次の作業者（AI・Human どちらでも）が気づける状態を維持するのが目的です。
