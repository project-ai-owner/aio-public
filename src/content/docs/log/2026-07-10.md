---
title: クラウドスケジューラが VM 認証情報に届かなかった
description: 開発ログ。クラウドトリガーは VM 上の認証情報にアクセスできないという制約を実証し、投稿ステップを VM cron に移行した記録。
---

**2026-07-10**

## 何が起きたか

AIO では、コンテンツ運用ケースの投稿ステップをクラウドスケジューラ（AI エージェントのルーティン）で実行しようとしていました。この日、それが動かないことが実証されました。

**原因**: クラウドセッションは AIO の VM 上に置かれた認証情報（投稿 API のキー）に到達できない。クラウドと VM は別の実行環境でファイルシステムを共有しておらず、クラウドから VM のクレデンシャルを読んで外部 API を叩くことは構造的にできない。

これは設計ミスではなく、前提の確認不足でした。

## どう直したか

投稿の実行ステップを VM cron に移行しました。

```
クラウドスケジューラ（毎日）
  → 記事の生成・Fact Review・PR 起案
  （投稿はしない）

VM cron（毎日夜）
  → daily_post.py を実行
  → Fact Review 通過済みの次候補を選定
  → 投稿 API を呼び出し（実行環境が認証情報と同じ場所にある）
  → 結果を書き戻し → git push
```

`daily_post.py` は冪等に設計しました。「今日投稿済みかどうか」を状態ファイルで判定し、二重投稿しない。停止スイッチ（ファイル 1 本置けば翌日以降の投稿を止める）も常置しています。

## 事後レビュー型への転換

この変更に伴い、運用パラダイムが変わりました。

**変更前（事前承認型）**: Human が確認 → 承認 → 投稿  
**変更後（事後レビュー型）**: VM cron が自動投稿 → Human が事後確認

「投稿の前に必ず Human が承認する」運用から、「Fact Review 通過・事前承認済みの候補ストックから VM が選んで投稿し、Human が事後チェックする」運用へ。最終責任の所在は変わらず、オペレーションのレイテンシが下がりました。

## Fact Review のスコープ確定

同日、AI が行う事実確認の範囲を明文化しました。

- **確認範囲**: 公開オンライン情報（公式サイト・公式ページ等、誰でも参照可能なもの）のみ
- **電話・取材・確認依頼は行わない**（Human も含む）
- **公開オンライン情報で確認できない事実は掲載しない**（確認不能 = 棄却）

「確認できないから掲載しない」と明文化することで、AI が曖昧な情報を「推定」で通過させることを防ぎます。運用コストを下げながら精度を担保する設計上の判断です。

## D-043 として採択

今日の変更（VM cron 移行・事後レビュー型・Fact Review スコープ）を Decision として記録しました。AIO では重要な方針変更を ADR（アーキテクチャ決定記録）として残し、後から「なぜそうなったか」を追跡可能にしています。

## 学び

**認証情報の局所性は自動化パイプラインの基本制約**。クラウドとオンプレ（または別の実行環境）では、クレデンシャルの置き場所が違う。この制約を設計段階で明確にしないと、「ここが届かない」という問題が後から出る。

今回の場合、「クラウドがコンテンツを準備し、VM が実行する」という役割分担に整理することで解決しました。両者を無理に統合しようとするより、それぞれの責務を明確に分けた方が安定します。
