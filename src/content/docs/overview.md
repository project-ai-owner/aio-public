## 背景

2026年6月末、AI エージェント（Claude Code）が「指示すればコードや文書を作れる」段階から「継続的な運用を部分的に任せられるか試せる」段階に入ったと判断し、個人プロジェクトとして開始しました。目的は最初から一貫しています: **AI と人間の協調による改善サイクルを、記録が残り・監査でき・止められる形で回すこと**。

## 解決したい課題

1. **セッションが揮発する**: チャット型 AI は会話が終われば状態を失う。長期プロジェクトには耐久記憶が要る
2. **権限の境界が曖昧**: 「AI がどこまで自律実行してよいか」を都度判断すると、人間がボトルネックになるか、逆に危険な操作が通る
3. **並列実行が衝突する**: 複数のエージェントやルーチンが同じファイルを触ると壊れる
4. **コストと利用枠の規律**: 個人の予算と AI サブスクリプションの利用枠を、無自覚に使い切らない配分規律が要る
5. **「完了した」の信頼性**: AI の完了報告と実際の状態は一致しないことがある。検証の仕組みが要る

## 人間と AI の役割

| 主体 | 役割 |
|---|---|
| **Human（プロジェクトオーナー）** | 最終意思決定（Decision 採択・方針転換・外部公開・課金/契約）・事後レビュー・停止権限。すべての権限の源泉 |
| **Claude Code（メインセッション）** | 実装・文書化・タスク分類と委任・PR 作成・条件を満たす場合の merge 代行・Human への報告と判断依頼 |
| **Claude subagents（4種）** | 記録系 / 事業運用系 / インフラ系 / レビュー系（read-only）にコンテキストを分離した委任先 |
| **定期ルーチン（クラウド実行）** | 日次監査・外部 AI 出力の取り込み・コンテンツ準備など、毎回ゼロから状態を再発見して働く使い捨てセッション |
| **ChatGPT** | 外部ディープリサーチ・独立レビューの依頼先（人間が仲介する疎結合連携。かつては merge 承認役も担ったが廃止 — [Lessons 01](../lessons/01-two-agent-retirement/)） |
| **GitHub** | 唯一の正（SSOT）。すべての状態・決定・変更履歴の置き場 |
| **Ubuntu VM** | 常時稼働の実行環境。認証情報が必要な処理はここだけで実行 |

## GitHub を SSOT とする理由

- AI のセッションは揮発するが、**PR・Decision・改善バックログに永続化すれば、どのセッションからでも状態を復元できる**
- 変更がすべて commit / PR / merge を通るため、AI の作業が**監査可能・巻き戻し可能**になる
- 人間のレビューゲート（PR 承認）を、権限設計の物理的な実装として使える

## 自動化の目的

自動化そのものが目的ではありません。**人間の仲介回数を減らし、人間の判断を「本当に人間にしかできない判断」に集中させる**ことが目的です。このため、権限を 3 層（事前承認済み / 条件付き / 人間でも解除できない禁止）に分け、層ごとに自動化の可否を決めています。

## 現状の限界

- **タスクキューの完全自動化は停止中**（急ぎすぎて手戻りが出たため。[Lessons 02](../lessons/02-automation-too-fast/)）
- **計測データの回収など、人間の手作業が残っている**箇所がある
- 高性能モデルは分析にのみ使い、実装は標準モデルで行う配分運用（利用枠の規律であり、必要なら課金拡張で解ける）
- 未完成の部分は [Current Status](../status/) で区分して公開しています

なお、VM のスペック（メモリ 2GB・並列 2）はパイロット段階の初期構成であり、限界ではなく選択です。実測で不足が確認された時点で拡張します。
