---

## 何が起きたか

AIO では改善項目の一覧（バックログ）や意思決定の索引（Decision ログ）のような**共有インデックスファイル**に、あらゆる作業が 1 行ずつ追記します。バックグラウンドエージェント・定期ルーチン・メインセッションが並列で動き始めると、この共有ファイルで PR 競合が発生しました。それぞれの PR は正しいのに、同じ行の近くを触るため rebase 地獄になる。加えて、完了報告と実際の git 状態が食い違う（エージェントは「反映した」と言うが、実際には別ブランチに取り残されている）ケースも重なりました。

## どう変えたか

1. **プレローンチ・チェックの義務化**: 並列タスクを起動する前に、各タスクの「担当ファイルリスト」を列挙し、重複がないこと（disjoint）を確認する。重複があれば直列化するかスコープを再設計する
2. **共有インデックスは直列化**: バックログ・Decision ログ・プロジェクト設定ファイルを変更するタスクは並列起動を禁止。定期ルーチンには「これらを触る open PR があれば、自分の変更を延期して報告だけせよ」と明記
3. **索引登録の延期パターン**: 本体ファイル（1 改善項目 = 1 ファイル）は先に merge し、共有インデックスへの行追加だけを後続 PR に延期できる運用を用意した
4. **完了報告を信用しない検証**: エージェント完了後に `git status` と PR 一覧を実際に確認する手順を標準化。「記録上の完了」と「検証済みの完了」を区別する

## 学び

- **ロック機構を作るより、衝突しない構造にする方が安い**。ファイルスコープの分離は、レビューも監査も単純にする
- 競合の根本原因は並列性ではなく、**「追記が集中するホットファイル」の存在**だった。1 項目 = 1 ファイルに分解し、インデックスは薄い索引に保つ設計変更が効いた
- AI エージェントの「やりました」は、人間の部下の報告と同じで、**検証プロセスとセットで初めて使える**。これは能力の問題ではなく運用設計の問題
